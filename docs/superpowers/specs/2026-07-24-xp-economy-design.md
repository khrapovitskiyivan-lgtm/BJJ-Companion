# Спек: XP-экономика и уровни («Duolingo поверх петли»)

Дата: 2026-07-24
Статус: черновик на согласование

## Контекст и цель

Спек геймификации 2026-07-16 реализован (6 статов, «Разрыв», порог холодного
старта, титул архетипа), но **XP, уровни и полоски прогресса были осознанно
отложены** («XP это экономика, а не фича — нужен отдельный дизайн»). Этот спек их
проектирует.

Стратегическая рамка — ставка РОСТ/удержание. Опрос (п.25.6) назвал главный риск
оттока: «прочитал всё за месяц-два → отписка» (истощение статичного контента).
Возобновляемая шкала прогресса лечит именно это. Но generic-счётчик — обёртка;
ценность появляется, когда прогресс **направлен** (растит 6 статов/архетип — моат
приложения) и **педагогичен** (бонус инвертирован по поясу).

**Это НЕ Дуолинго.** Обучение происходит на мате 3-4 р/нед по расписанию тренера,
приложение — компаньон/дневник. Отсюда явные отказы (ниже): нет daily-стрика в XP,
нет лиг, нет «жизней», нет магазина.

## Принципы

1. **Одна валюта, один уровень игрока.** Не 6 параллельных XP-шкал (плотность
   «Моей игры» на пределе, п.28.C).
2. **XP только из ядра петли** — реального сигнала (лог тренировки), не из «зайди в
   приложение» и не из самоотметки «изучил» в библиотеке (иначе ферма XP; XP мерит
   активность, % прогресса мерит знание — две оси раздельно, как в остальном
   приложении).
3. **XP выводится, не хранится** (см. Архитектура) — как `stats.ts`/`reward.ts`.
4. **Никакого распада и штрафов** — XP только копится; отдыхом прогресс не теряешь.
5. **Простота и обратимость** — не зайдёт по телеметрии, удаляем модуль + UI-хуки,
   данные не тронуты.

## Формула XP

Считается по записям дневника (тот же принцип, что `computeEntryReward`).

| Источник | XP | Кап на запись |
|---|---|---|
| Запись тренировки в дневник | +20 | база |
| Техника в записи | +10 каждая | 3 (max +30) |
| Поясной бонус за технику «в направлении роста» | +10 каждая | 3 (max +30) |
| Разобрал показанное (открыл карточку из очереди) | +10 каждый | 2/день (max +20) |

Типичная запись ≈ 60 XP, большая тренировка ≈ до 100.

### Поясной бонус (инверсия по поясу)

Направление роста зависит от пояса (принято в спеке 2026-07-16, п.6):

- **Белый пояс** — бонус за технику, стили которой НЕ пересекаются с топ-архетипом
  игрока (охват базы, широта).
- **Синий и выше** — бонус за технику, стили которой пересекаются с топ-архетипом
  (специализация, глубина).

Топ-архетип — `computeStyleAffinity(progress, practiceCount)`, первый по score.
Считаем по **текущему** поясу и **текущему** архетипу (смена пояса редка; точность
«пояс на момент лога» не стоит хранения истории).

### Отметка «изучил»/«в процессе» в библиотеке — 0 XP

Самоотметка не даёт XP осознанно (защита от фарма, разведение осей XP vs знание).
Обучение отражается в % прогресса и скилах, XP — за реальные тренировки в дневнике.

## Пороги уровней

Стоимость перехода: **cost(L → L+1) = min(50 × L, 400)**.

- 1→2: 50 (первая же запись поднимает уровень — победа в первой сессии)
- 2→3: 100, 3→4: 150, … 8→9: 400, дальше плато 400.

При ~3 тренировках/нед (~180 XP/нед): ~уровень 6 за первый месяц, потом ~1 уровень
в 2 недели. Частые ранние победы, осмысленный рост дальше.

Уровни — **просто числа**, без титула на каждый (идентичность несёт титул архетипа,
уже есть; имена десяткам уровней — контент-долг).

## Скилы (6 статов как уровни)

6 статов уже считают `pct` 0..100 (`stats.ts::computeStats`). Скил-уровень —
**производная от pct** (бакеты), НЕ отдельный XP-пул:

`skillLevel(pct) = Math.max(1, Math.floor(pct / 10))` → 1..10 (без «ур. 0»).

Это переосмысление существующего % в «ур. N» для читаемости направления роста.
Никакой новой прокачки — только презентация в аккордеоне «Характеристики».

## Архитектура: XP выводится

Новый чистый модуль `src/lib/bjj/xp.ts` (образец — `reward.ts`/`stats.ts`):

```
computeTotalXp(input: {
  entries: DiaryEntry[];
  progress: ProgressMap;
  belt: Belt;
  techniques: Technique[];
  reviewed: Set<number> | Record<number, ...>; // из useReviewed
}): number

levelForXp(totalXp: number): {
  level: number;
  xpIntoLevel: number; // XP, набранные внутри текущего уровня
  xpForLevel: number;  // стоимость текущего уровня (для полоски)
  xpToNext: number;
}

skillLevel(pct: number): number // 0..10
```

Почему выводится, а не хранится:

- **Ретроактивность бесплатно** — у существующих пользователей уже есть история
  дневника, в день релиза они на честном уровне (не «полгода веду дневник, 1 уровень»).
- **Нет мутируемой валюты, нет конфликтов синхронизации** — derive из уже
  синхронизируемых `entries`/`progress`.
- **Обратимость** — удаление модуля не трогает данные.
- Паттерн «дельта до/после» для экрана награды уже есть в `reward.ts` (pctExact
  before/after) — XP-дельту считаем так же.

**Нюанс:** компонент XP «разбор показанного» использует `useReviewed`, который
локальный без облака (по решению п.27, UI-нюанс). Значит на свежем устройстве эта
малая часть XP пересчитается ниже, пока карточки не открыты снова. Принимаем: вклад
мал, приложение в основном одно-девайсное (Telegram на телефоне), диаметр дрейфа
незначителен. Основная масса XP (лог + per-technique + поясной бонус) — из
синхронизируемого дневника.

## Поверхности (4 существующие, новый экран не заводим)

### 1. Верх «Моей игры» — статус на виду (`ProgressTop.tsx`)
Бейдж уровня на аватаре профиль-шапки + тонкая полоска XP до следующего в строке.
Единственная постоянно видимая точка прогресса. Тап по шапке уже открывает лист
игрока.

### 2. Экран награды — где XP начисляется (`EntryReward.tsx` + `reward.ts`)
Добавляем в `EntryReward`:
- «+N XP» суммой сверху;
- разбивка по действиям (запись / техники / поясной бонус золотой строкой / разбор);
- полоска уровня внизу; при переходе уровня — момент level-up в той же шторке.

`computeEntryReward` расширяется полем `xp` (или отдельная функция `computeEntryXp`,
вызываемая рядом) с дельтой XP и флагом level-up. Числа — по формуле выше.

### 3. «Характеристики» (аккордеон, свёрнут) — направление роста
6 статов показывают «ур. N» из `skillLevel(pct)` рядом со шкалой. Без отдельных
XP-полосок. Плотность не растёт.

### 4. Лист игрока (`CharacterSheet.tsx`) — витрина
Уровень, XP до следующего, разбивка XP по источникам за всё время (опционально),
титул архетипа (уже есть).

## Партнёры (`partnersProfile.ts`, `PartnersBlock.tsx`, api + SQL)

Геймификация отражается и в партнёрах — как **идентичность**, НЕ как лидерборд.

- **Строка партнёра в списке** — маленький бейдж уровня рядом с именем/поясом.
- **Карточка партнёра** — число уровня рядом с титулом архетипа + 6 скилов-уровней
  (партнёр выводит их из уже публикуемого `stats` pct, `skillLevel(pct)`).
- **Полоску «XP до следующего» у чужого профиля НЕ показываем** (только своё).
- **Сортировку списка НЕ трогаем** — остаётся «отстающие по недельному плану сверху».
  Уровень — не ключ сортировки.

### Главный запрет (экспертный)
Партнёры НЕ становятся лидербордом по XP: сравнение уровней демотивирует (занятой
новичок против ветерана), домен добровольный. Уровень = контекст личности, не
соревновательный счёт. Никаких «ты обошёл партнёра на N XP».

### Приватность
Уровень отражает тренировочную активность (XP = дневник), а НЕ число изученных
техник (которое решено не раскрывать). Показывает ровно то, что подотчётность и так
подразумевает. Согласуется с «наружу только агрегаты».

### Данные
- `buildPublishInput` включает вычисленный **уровень** игрока (один новый скаляр).
  Скил-уровни партнёр деривит из уже публикуемого `stats` — новых полей под них нет.
- Новая колонка `level int` в `bjj_partner_profiles` + возврат в RPC списка/карточки.
  SQL применяет пользователь (образец — прошлые миграции партнёров).
- Клиент партнёров уровень не пересчитывает — показывает пришедшее с сервера.

## Телеметрия

- `level_up` (detail = новый уровень) — на переходе уровня (дедуп по уровню/сессии).
- Расширить whitelist `bjj_track` отдельным `docs/sql/2026-07-24-telemetry-xp.sql`.
- Специальных событий начисления XP не заводим (шумно; уровень достаточно).

## Не входит (YAGNI / отложено)

- Daily-стрик в XP, лиги, «жизни», магазин/валюта.
- Титулы на каждый уровень.
- «Партнёр вышел на N уровень» как нудж холодного старта — отдельным шагом по данным.
- Воскресный recap бота «+N XP за неделю» — лёгкий довесок, можно добавить позже
  переиспользованием канала recap; в ядро этой итерации не входит.
- XP за отметку «изучил» — сознательный отказ (см. выше).

## Порядок реализации (маленькие проверяемые циклы)

1. `xp.ts` (чистый, TDD): computeTotalXp + levelForXp + skillLevel. Тесты на формулу,
   пороги, поясную инверсию, капы.
2. XP-дельта в `reward.ts` (TDD) + блок «+N XP»/level-up в `EntryReward`.
3. Бейдж уровня + полоска XP в `ProgressTop`; скилы-уровни в «Характеристиках»;
   витрина в `CharacterSheet`.
4. Партнёры: level в publish + SQL-колонка + бейдж/карточка (после проверки own-стороны).
5. Телеметрия `level_up` + SQL whitelist.

## Открытые допущения (сверить при ревью)

- Числа формулы (20/10/10/10, капы 3/3/2) и пороги (50×L, плато 400) — стартовые,
  калибруются по телеметрии; менять дёшево (константы модуля).
- `skillLevel = floor(pct/10)` — простой бакет; при желании кривее.
- Поясной бонус по текущему поясу/архетипу — принятое упрощение derive.
