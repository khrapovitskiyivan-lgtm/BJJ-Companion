# Спек: геймификация «Моей игры»

Дата: 2026-07-16
Статус: черновик на согласование

## Цель

Превратить раздел «Моя игра» из статистики в игровой профиль, который **ведёт**
белого пояса, а не просто размечает. Ядро ценности — не XP и статы (это обёртка),
а фича «Разрыв»: показать расхождение между тем, кем игрок хочет быть, и тем, что
он реально тренирует по дневнику. Этого не может ни справочник, ни YouTube — только
дневник, потому что нужны реальные записи за недели.

## Принцип: две независимые оси

Главное архитектурное решение. Чтобы «профиль статов архетипа» не был перекрашенным
архетипом, статы и архетип считаются из **разных полей**:

- **Архетип** — из `technique.styles` (что играешь, стратегия). Уже реализовано в
  `styleProfile.ts::computeStyleAffinity`.
- **6 статов** — из `technique.tags` (как это работает, механика).

Разные источники → разный сигнал → модель не схлопывается.

## Данные (проверено, переразметка CSV не нужна)

Все поля уже заполнены в `generated/techniques.json` (293 техники):
- `styles` — 1-3 архетипа на технику.
- `tags` — 78 механических тегов.
- `points_ibjjf`, `legal_ibjjf_gi`, `legal_adcc` — для цели «соревнования».
- `group`, `belt`, `difficulty`.

### 6 статов из тегов

| Стат | Теги | Покрытие |
|---|---|---|
| Контроль | angle_control, control | 140 |
| Давление | pressure, weight_distribution | 118 |
| Структура | frames, base_break | 103 |
| Рычаг | limb_isolation, joint_lock, leverage | 102 |
| Гибкость | flexibility | 51 |
| Скорость | speed | 34 |

Стат техники: есть ли у неё хотя бы один тег из набора (булево, не сумма).

### Профили архетипов выводятся из данных, не пишутся руками

primary/secondary стат каждого архетипа считается через **lift** = доля техник со
статом внутри архетипа / доля этого стата по всей базе. Проверено — даёт осмысленный
результат (Прессинг-пассер → Давление, Игра ног → Рычаг, Борец → Скорость). Таблица
генерируется скриптом сборки, чтобы не расходиться с CSV.

## Компоненты

### 1. Цель → рекомендации (чинит мёртвое поле)

Сейчас `profile.goal` пишется онбордингом и **не читается никем** (проверено). Это
невыполненное обещание, чинится первым и ни от чего не зависит.

`nextToLearn` в `recommend.ts` получает мягкий бонус к сортировке по цели:
- `self-defense`: приоритет `group` in {escape, takedown} + `fundamental` тег.
- `competition`: приоритет `points_ibjjf > 0` и `legal_ibjjf_gi` (по формату профиля).
- `hobby`: усилить существующий round-robin по группам (разнообразие).

Бонус к весу сортировки, не жёсткий фильтр — иначе сузим выдачу.

### 2. Шесть статов в «Моей игре»

Новая секция под hero-статистикой. Для каждого стата:
- значение = % освоенных техник этого стата от всех техник этого стата в базе
  (проценты, не сырьё — иначе Скорость с 34 техниками вечно пустая).
- вес техники как в `computeStyleAffinity`: done=2, in_progress=1, дневник +1.5.

Функция `computeStats(progress, practiceCount)` в новом `src/lib/bjj/stats.ts`.
Профили архетипов (primary/secondary) — сгенерированная константа.

### 3. Порог холодного старта

До 5 освоенных техник архетип и разрыв не показываем — выводим «Стиль определяется,
отметь ещё N техник». Иначе игрок получает ярлык от двух галочек в опросе (наблюдалось
вживую: 2 техники → архетип 100%).

### 4. Аспирация vs архетип

Развести два существующих источника правды по смыслу:
- `profile.preferredStyles` (руками в настройках) = **аспирация**, «кем хочу быть».
- `computeStyleAffinity` (из прогресса) = **архетип**, «кто я есть».

### 5. Фича «Разрыв» (ядро)

Карточка в «Моей игре», видна только когда: задана аспирация И пройден порог из п.3.

Показывает:
- аспирационный стиль и его реальный % в игре;
- топ реальный архетип и его %;
- если совпадают — «Идёшь по плану»;
- если расходятся — «Хочешь X (aspiration), тренируешь Y. Ближайшие 3 техники стиля
  X:» + список + кнопка «Собрать тренировку» → генератор по дневнику (уже готов).

Данные: `computeStyleAffinity` (архетип) + `preferredStyles` (аспирация) +
`nextToLearn`, отфильтрованный по техникам аспирационного стиля.

### 6. Титулы — в этой итерации; XP — отложен

- **Титул архетипа** (косметика) делаем сейчас: архетип уже вычисляется, показать
  «Прессинг-пассер» как титул у аватара — дёшево.
- **XP отложен на следующую итерацию** (решение 2026-07-16): это экономика, а не
  фича — нужен отдельный дизайн (за что капает, что такое уровень). Сначала статы
  и разрыв проверят, заходит ли игровой профиль вообще. Принятая на будущее
  механика: бонус **инвертирован по поясу** — белый пояс получает бонус за техники
  ВНЕ своего архетипа (охват базы), синий+ — за свои (специализация).
- **Ни одна техника не прячется.** Отказ от `unique techniques`: приложение —
  справочник/дневник, спрятать реальную технику ломает основную работу и конфликтует
  с free/paid. Архетип выделяет («техники твоего стиля»), а не запирает.

### 7. Аватар персонажа

Схема v2 (2026-07-16): составной аватар из 15 тел (body-{kimono}-{belt}) и 12 голов
(head-{m1..f6}, прозрачный фон), итого 180 комбинаций. В профиле {kimono, headId};
пояс тела берётся из profile.belt на лету. Онбординг: шаг выбора головы вместо пола.

- Новое поле в `StyleProfile`, отдельно от `avatarUrl` (там фото из Telegram — остаётся).
- Разместить в «Моей игре» четвёртой карточкой в ряду hero-статистики.
- Кружок из Telegram в шапке остаётся на месте.

## Порядок реализации

Каждый пункт — отдельный маленький цикл, проверяемый в рантайме:

1. Цель → рекомендации (чинит существующее, независимо).
2. 6 статов + порог архетипа (данные готовы).
3. Разрыв (требует 1 и 2) — ядро.
4. Аватар (нарезка + выбор пола) + титул архетипа.

## Не входит в этот спек (YAGNI / отдельно)

- XP-экономика, уровни персонажа, полоски прогресса (отложено — см. п.6).
- Список клубов/команд (из идеи 11) — отдельный подпроект.
- Roadmap-вид графа — закрыт: это зональная раскладка Focus Mode, отдельного вида нет.
