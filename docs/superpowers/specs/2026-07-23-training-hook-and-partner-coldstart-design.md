# Дизайн: крючок после тренировки + вечерний пинг + холодный старт партнёров

Дата: 2026-07-23
Статус: согласован (брейншторм), ждёт плана реализации.

## Контекст и мотивация

Из живого опроса (CLAUDE.md п.25): главная боль удержания — «показали на тренировке, но забыл, выйдя с зала», и разрыв «понимание -> тело». Крючок «зачем открыть завтра» = не потерять показанное + задрилить самому. Тренер выбирает тему на месте и не предупреждает, поэтому крючок строится ПОСЛЕ класса: записал что показали -> разобрал (видео/детали/связи) -> закинул в свою отработку.

Параллельно (CLAUDE.md п.24, след. сессия) — холодный старт партнёров: проактивный промпт «позови партнёра по залу» после первой недели в плане/серии, чтобы включить взаимную подотчётность у пользователей с темпом, но без партнёров.

Документ покрывает две независимые, но родственные (обе про удержание) части. Обе шипятся вместе.

## Часть A. Петля вокруг тренировки

Три компонента: постоянный блок «Разбери показанное» (сердце), быстрый путь на экране награды, вечерний пинг-нудж.

### A1. Store `useReviewed` и функция `pendingReview`

Новое локальное состояние: какие показанные техники пользователь уже «разобрал».

- Хук `useReviewed` в `src/lib/bjj/store.ts`. Ключ localStorage `bjj.reviewed.v1`, тип `Record<number, number>` (id техники -> момент разбора в ms). Схема ровно как у `useNotes`: module-level снимок + `Set` подписчиков (общая шина), чтобы запись из карточки техники дошла до блока на «Моей игре» без перемонтирования.
- **Без облачной синхронизации.** Это UI-нюанс очистки блока, а не пользовательские данные; колонка в Supabase не нужна. Локально, per-device.
- API хука: `{ reviewed, markReviewed(id), hydrated }`. `markReviewed(id)` пишет `reviewed[id] = Date.now()`.

Чистая функция в новом модуле `src/lib/bjj/reviewQueue.ts`:

```
pendingReview(
  entries: DiaryEntry[],
  reviewed: Record<number, number>,
  progress: ProgressMap,
  today: Date,
  windowDays = 7,
  cap = 6,
): number[]
```

Логика:
1. Для каждой техники собрать `shownAt` = максимальная дата записи (как локальная полночь дня, ms), среди записей с `date` в окне `[today - windowDays + 1 .. today]`. Берём только `techniqueIds` (НЕ `caughtBy`).
2. Пропустить технику, если `progress[id] === "done"` — освоенное не нуждается в «разборе» (см. ниже).
3. Техника попадает в очередь, если `reviewed[id]` не задан ИЛИ `reviewed[id] < shownAt` (т.е. её не открывали после последнего лога).
4. Сортировка по `shownAt` убыв. (свежие сверху), затем по id для детерминизма. Обрезать до `cap`.

**Изученные (`done`) исключаются.** Очередь — только край обучения (`not_started`/`in_progress`). Боль опроса — «показали новое, забыл»; `done` пользователь сам отметил освоенным, нудж «разбери» для него шум (тем более онбординг засевает знакомые техники как `done`). Практика совпадает: `diary.tsx::save` поднимает `not_started -> in_progress`, но `done` не трогает — свежепоказанное новое попадает, освоенное отсеивается. Свойство «выпуска»: пометил `done` -> техника перестаёт всплывать.

Свойство «повторный лог возвращает технику»: сравнение идёт с датой ПОСЛЕДНЕГО лога, а не «разобрано навсегда». Открытие карточки ставит `reviewed = Date.now()` (сегодня, после полуночи) — покрывает лог сегодняшнего дня. Повторный лог через неделю даёт `shownAt` новой полуночи `> reviewed` -> техника снова в очереди (если к тому моменту не `done`).

Тесты (`reviewQueue.test.ts`): в окне/вне окна; разобрано (открыл карточку) уходит; повторный лог возвращает; изученная (`done`) не попадает; кап и порядок; `caughtBy` не попадает.

### A2. «Разобрал» = открыл карточку

На маршруте `src/routes/technique.$id.tsx`: в `useEffect` по id вызвать `markReviewed(id)` (после гидратации `useReviewed`). Глобально — любое открытие карточки помечает технику разобранной, откуда бы ни пришли (блок, экран награды, поиск). Побочных эффектов нет: пометка влияет только когда техника позже окажется в очереди.

### A3. Блок «Разбери показанное» на «Моей игре»

Новый компонент `src/components/bjj/ReviewShownBlock.tsx`, размещается на `src/routes/progress.tsx` сразу под `TodayCard` (момент времени-чувствителен: разбирать, пока свежо).

- Данные: `useDiary`, `useReviewed`, `useProgress`; `today = new Date()` только после гидратации (SSR-гард как в `TodayCard`).
- `const ids = pendingReview(entries, reviewed, progress, today)`. Если `ids.length === 0` — компонент возвращает `null` (само-очищается, плотность не растёт).
- Разметка: заголовок-секция «Разбери показанное» + короткий кикер-пояснение (например «Открой карточку — освежи детали и связи»). Список строк техник через существующий `TechniqueRow` (из `TechniqueCard.tsx`), каждая — `Link` на `/technique/$id` (открытие пометит разобранной). Внизу — одна soft-кнопка «В отработку» -> `Link` на `/workout?src=diary`.
- Телеметрия: `review_opened` при тапе по технике (detail = id), `review_drill` при тапе «В отработку».

Почему `?src=diary`, а не засев конкретными id: `generateWorkoutFromDiary` уже приоритезирует техники «в процессе» и давно не отработанные; сохранение записи ставит показанные техники «в процессе» (`diary.tsx::save`), поэтому они и так всплывают в отработке по дневнику. Отдельный режим генератора «фокус на этих id» — избыточная сложность (YAGNI).

### A4. Экран награды — быстрый путь

В `src/components/bjj/EntryReward.tsx` под каскадом дельт добавить секцию «Разбери показанное»:
- Строки залогированных техник (`TechniqueRow` как `Link` на карточку; тап = разобрано и уводит с награды).
- Тот же фильтр `done`: показываем только не-освоенные из записи; если после фильтра пусто — секция не рендерится.
- Одна soft-кнопка «В отработку» -> `/workout?src=diary`.
- Проп `techniqueIds: number[]` в `EntryRewardSheet` — уже отфильтрованный от `done`. Фильтр делается в `diary.tsx::save`, где есть `progress`: `picked.filter((id) => (progress[id] ?? "not_started") !== "done")` (на данных «до», как и расчёт награды). Передать вместе с `reward` (расширить state c `EntryReward | null` до `{ reward, techniqueIds } | null`). Сам `EntryRewardSheet` `progress` не получает — приходит готовый список.

Это дешёвый путь «разобрал сразу»; постоянный блок A3 ловит тех, кто закрыл награду не разобрав.

### A5. Вечерний пинг (soft-нудж)

Расширяем существующий крон (`api.tg-cron.ts`, 20:00 МСК) и чистую `decide()` (`tgRemind.ts`). Крон видит только `bjj_tg_chats` (тренировочные дни, `last_entry`, недельный план) — НЕ содержимое дневника. Значит пинг = нудж «поймать показанное» (записать); сам разбор идёт в блоке A3.

Новый результат `decide()`: `kind: "soft"`. Приоритет (одно сообщение в день, `last_ping` уже кэпит):
1. `muted` / нет `frequency` / `last_ping === today` / stale (>21 дн) -> `none` (как есть).
2. Воскресенье -> `recap` (как есть).
3. Не тренировочный день -> `none`.
4. Окно недели прошло (`after === 0`, сегодня последний трен.день) -> `none` (как есть).
5. `last_entry === today` (уже логировал) -> `none`.
6. `need >= after` (план горит) и `need > 0` -> `remind` (срочный, как есть).
7. Иначе трен.день + `need > 0` (недобор) + soft-кап не исчерпан на этой неделе -> `soft`. Иначе `none`.

Кап soft-нуджей: `SOFT_CAP = 2` в неделю. Две новые колонки `bjj_tg_chats`:
- `soft_ping_week date` — понедельник недели последнего soft-нуджа.
- `soft_ping_count int default 0`.

В `decide()` доступное число: `usedThisWeek = row.soft_ping_week === mondayIso ? row.soft_ping_count : 0`. Разрешаем `soft`, если `usedThisWeek < SOFT_CAP`.

В кроне при УСПЕШНОЙ отправке:
- `kind === "soft"`: PATCH `last_ping = today`, `soft_ping_week = mondayIso`, `soft_ping_count = (row.soft_ping_week === mondayIso ? row.soft_ping_count + 1 : 1)`.
- `remind`/`recap`: PATCH только `last_ping = today` (как есть).

`decide()` получает `mondayIso` (уже передаётся). Поля `soft_ping_week`/`soft_ping_count` добавить в `TgChatRow`.

Текст soft: `Была тренировка сегодня? Запиши, что показали — потом разберёшь в приложении. Отключить: /mute`. Инлайн-кнопка «Открыть дневник» (`/diary`, как у remind).

SQL-миграция `docs/sql/2026-07-23-evening-ping.sql`: `ALTER TABLE bjj_tg_chats ADD COLUMN soft_ping_week date, ADD COLUMN soft_ping_count int NOT NULL DEFAULT 0;`. Применяет пользователь.

Тесты (`tgRemind.test.ts`): soft на трен.дне при недоборе и не-горящем плане; soft НЕ летит при исчерпанном капе; приоритет remind над soft (план горит); soft НЕ летит если логировал сегодня; сброс счётчика на новой неделе.

## Часть B. Холодный старт партнёров

Проактивный призыв «позови партнёра по залу», когда у пользователя есть темп, но нет партнёров.

### B0. Почему без отдельного компонента

`PartnersBlock` УЖЕ рендерится на `progress.tsx` сразу под `TodayCard` (progress.tsx:148-150) и уже всегда показывает приглашение: заголовок «Партнёры», empty state «Пока никого / Пригласи партнёра», кнопку «Пригласить» в шапке. Отдельный `PartnerNudge` встал бы вплотную над блоком и продублировал его. Поэтому холодный старт встраиваем В empty state самого `PartnersBlock` (momentum-aware), нового компонента нет.

### B1. Momentum-aware empty state в `PartnersBlock`

Правки в `src/components/bjj/PartnersBlock.tsx`, только в ветке `partners.length === 0` (когда `enabled = inTg && hasConsent()`):
- Посчитать темп из уже доступных в компоненте `entries`/`profile`: `weeksStreak = planStreak(trainedByDate(entries), frequency, today)` при заданной частоте, иначе `dayStreak(trainedByDate(entries), today)`. `hasMomentum = weeksStreak >= 1 || dayStreakNoPlan >= 3`. (`today = new Date()`; компонент клиентский, `enabled` уже гейтит TG.)
- Если `hasMomentum` — вместо тихого `EmptyState` показать активный призыв: строка «Ты в темпе (серия N недель / N дней подряд) — держать план проще с партнёром по залу» + primary-кнопка «Позвать партнёра» (тот же `onInvite`). Формулировку про серию строить по тому же плюрализатору, что в проекте.
- Если НЕ `hasMomentum` — прежний `EmptyState` («Пока никого / Пригласи партнёра»), без изменений.

Гейт по темпу принципиален: активный призыв не летит холодному пользователю, которому нечего показать партнёру; ему остаётся прежний тихий хинт.

### B2. Общий invite-хелпер и телеметрия

- Вынести invite-флоу в общий хелпер, чтобы momentum-кнопка и обычная «Пригласить» не дублировали логику. Хелпер `sharePartnerInvite(input: PublishInput): Promise<string | null>` в `partners.ts`: `publishProfile(input)` -> при коде `markPartnersJoined()` -> `shareText(текст, buildInviteLink(code))` -> вернуть код. `PartnersBlock.onInvite` переходит на него (тост об ошибке и `track` остаются в компоненте — у него есть UI-состояние).
- Короткий инвайт: одна кнопка сразу открывает штатную шторку выбора чата Telegram (как сейчас в `shareText`), без промежуточных экранов.
- Телеметрия: событие `partner_nudge` (без detail) на КЛИК momentum-кнопки — чтобы отличать инвайты из холодного старта от инвайтов кнопкой в шапке. Обычная «Пригласить» шлёт только `invite_created` (как есть). Оба пути также шлют `invite_created` внутри флоу.

Второй сурфейс для B (например, на экране награды) сейчас НЕ делаем (YAGNI). Если телеметрия `partner_nudge` покажет низкую конверсию — добавим момент по данным.

Локального флага `bjj.partnerNudge.v1` НЕ вводим: после приглашения `isPartnersJoined()` и так уводит из empty state; отдельный dismiss не нужен (empty state и так исчезает, как только появился партнёр).

## Общее

### Телеметрия

Расширить white-list RPC `bjj_track` тремя событиями: `review_opened`, `review_drill`, `partner_nudge`. SQL `docs/sql/2026-07-23-telemetry.sql` (create or replace функции с обновлённым списком), применяет пользователь. До применения события глотаются (как и раньше) — не ломает.

### Правила проекта

- Без эмодзи в исходнике (сырой эмодзи ломает iOS JavaScriptCore) и без em-dash в коде/текстах.
- Правило языка цвета: золото строго у достижений/тепла; статусы серый -> `--status-progress` -> зелёный. Новые блоки не вводят новых смыслов цвета.
- Типографика: кикеры `text-[11px] uppercase tracking-widest`, секции `text-sm font-semibold` — как в проекте.
- Хирургические правки, стиль окружающего кода. Общие контролы из `ui.tsx` (`Button`, `Section`, `EmptyState`), строки техник — `TechniqueRow`/`TechniqueChip`.
- Проверка рантаймом в обеих темах до слов «готово» (зелёная сборка не гарантирует webview). Крон/`decide` — юнит-тестами; ручную e2e-доставку пинга проверяет пользователь на реальном чате.

### Затрагиваемые файлы

Новые: `src/lib/bjj/reviewQueue.ts` (+ тест), `src/components/bjj/ReviewShownBlock.tsx`, `docs/sql/2026-07-23-evening-ping.sql`, `docs/sql/2026-07-23-telemetry.sql`.
Правки: `store.ts` (`useReviewed`), `technique.$id.tsx` (`markReviewed`), `progress.tsx` (блок `ReviewShownBlock` под Today), `EntryReward.tsx` + `diary.tsx` (проп `techniqueIds`), `tgRemind.ts` + `api.tg-cron.ts` (soft-пинг), `partners.ts` (`sharePartnerInvite`; `isPartnersJoined` уже есть), `PartnersBlock.tsx` (momentum-aware empty state + переход на общий хелпер), `telemetry.ts` (типы событий).

### Порядок реализации

A и B независимы. Рекомендуется A (ядро удержания) первым, затем B. Внутри A: A1/A2 (store + очередь) -> A3/A4 (UI блока и награды) -> A5 (пинг). SQL-миграции пользователь применяет до деплоя серверной части (A5, телеметрия).

- Второй сурфейс холодного старта партнёров (по данным телеметрии позже).
- EN-локализация новых строк (общий бэклог: EN в последнюю очередь).
